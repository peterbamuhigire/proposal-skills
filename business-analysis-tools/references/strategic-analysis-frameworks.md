# Strategic Analysis Frameworks

Frameworks for assessing organisational state, diagnosing problems, identifying root causes, and comparing current performance against desired outcomes or external benchmarks. These tools typically feature in Phase 1 (Assessment/Diagnostic) of a consulting methodology.

---

## Situation Analysis

### SWOT Analysis

Assesses internal strengths/weaknesses and external opportunities/threats.

| | Positive | Negative |
|---|---|---|
| **Internal** | Strengths (brand reputation, skilled staff, market share, processes) | Weaknesses (low recognition, low capitalisation, skill gaps, outdated systems) |
| **External** | Opportunities (underserved markets, policy changes, technology trends, competitor gaps) | Threats (increased competition, new regulations, economic downturn, technology disruption) |

**Best practice:** Never present a SWOT alone — always follow with "So what?" analysis:
- Which strengths to leverage
- Which weaknesses to address (and how)
- Which opportunities to prioritise
- Which threats to mitigate (and contingency actions)

**Proposal language:** "Our assessment will begin with a SWOT analysis of the [institution/sector], systematically mapping internal capabilities against external dynamics."

### PESTLE Analysis

Environment scanning across six dimensions.

| Factor | Description | Typical Assignments |
|---|---|---|
| **Political** | Government policy, political stability, regulatory direction | Public sector reform, governance |
| **Economic** | Growth rates, inflation, exchange rates, fiscal environment | Financial sector, economic development |
| **Social** | Demographics, cultural factors, health, education levels | Health, education, social protection |
| **Technological** | Digital infrastructure, technology adoption, innovation | ICT, digital transformation |
| **Legal** | Legislation, regulatory frameworks, compliance requirements | Regulatory reform, legal framework |
| **Environmental** | Climate, natural resources, environmental regulations | Agriculture, energy, infrastructure |

**Proposal language:** "We will apply a PESTLE analysis to map the external environment shaping [sector/institution] performance."

## Diagnostic and Assessment Tools

### Gap Analysis

Compares current state to desired state. The gaps drive the recommendations.

| Area | Current State | Desired State | Gap | Priority | Recommended Action |
|---|---|---|---|---|---|
| [Area 1] | [Description] | [Target] | [Difference] | [H/M/L] | [Action] |

**How it works:**
1. Assess current state through data collection, observation, document review
2. Define desired state from ToR objectives, international best practice, regulatory requirements
3. Identify gaps between the two
4. Prioritise gaps by impact and feasibility
5. Develop recommendations to close priority gaps

**Proposal language:** "Phase 1 will establish the current state through [methods]. Phase 2 will benchmark against [standard]. The gap analysis will form the basis for our recommendations in Phase 3."

### Benchmarking

Comparison of metrics, processes, or performance against reference points.

| Type | Compares Against | Example |
|---|---|---|
| **Internal** | Other units within the same organisation | Branch-to-branch performance comparison |
| **Competitive** | Direct peers or competitors | University-to-university administrative cost comparison |
| **Functional** | Best practice in a specific function, regardless of sector | Comparing procurement processes against ISO 20400 |
| **International** | Peer institutions in other countries | Comparing Uganda's revenue authority against Kenya's and Rwanda's |

**Methods:** Casual review (public documents, web pages), study of major variables / "secret shopping", reverse-engineering a specific product or process.

**Deloitte pattern:** Reference proprietary benchmark databases as a tool the firm brings — not just experience but comparative data.

**Proposal language:** "We will benchmark the client's [process/function] against [N] comparator [organisations/countries] to identify performance gaps and transferable good practices."

### Maturity Model Assessment

Structured assessment of how advanced an organisation's practices are in a specific domain.

| Level | Name | Description |
|---|---|---|
| 1 | **Initial** | Processes are ad hoc and reactive |
| 2 | **Managed** | Basic processes established, but not standardised |
| 3 | **Defined** | Processes documented and standardised across the organisation |
| 4 | **Measured** | Processes monitored with metrics and performance data |
| 5 | **Optimised** | Continuous improvement embedded, processes regularly refined |

Adapt the levels and naming to the specific domain (e.g., IT maturity, data governance maturity, gender mainstreaming maturity, procurement maturity).

**Proposal language:** "We will assess [institution] using a five-level maturity model adapted to the [domain] context, establishing a baseline and defining a target maturity trajectory."

### Feasibility Analysis

Evaluates whether a proposed solution is viable across multiple dimensions.

| Factor | Key Questions |
|---|---|
| **Operational Feasibility** | How well does it meet business needs? How does it fit the organisation? Sustainability, maintainability? |
| **Technology/System Feasibility** | Does the technology exist? Can it be acquired? Does the organisation have technical expertise? |
| **Cost-Effectiveness Feasibility** | What are up-front and ongoing costs? Are solutions affordable vs. expected benefits? |
| **Time Feasibility** | Can it be delivered within time constraints? Can it be staged? |

## Root Cause Analysis

### Five Whys
- Ask "why" up to five levels deep to reach the true root cause
- Avoid using literal "why" — rephrase as appropriate questions
- Good starting point before other techniques
- **Proposal language:** "We will apply iterative root cause analysis techniques to move beyond symptoms to underlying systemic issues."

### Fishbone (Ishikawa) Diagram
- Problem at the head, causes as bones
- Use 3–8 categories (People, Process, Technology, Policy, Environment, Materials, Measurement, Methods)
- Go 2–3 levels deep for subcauses
- Look for patterns and circle significant factors

### Interrelationship Diagram
- Depicts complex relationships among multiple variables (up to 10 factors)
- Arrows show direction of causation
- Factors with most "incoming" arrows = key effects (good measures of success)
- Factors with most "outgoing" arrows = root causes (prioritise these)

### Pareto Analysis (80/20 Rule)
- Identify the 20% of causes producing 80% of the problems
- Visualise as a Pareto diagram (bar chart sorted by frequency with cumulative line)
- Focus intervention on the vital few causes

### Affinity Diagram
- Organise ideas into categories and subcategories that cluster together
- Useful for connecting related issues and generating needed capabilities
- Good complement to brainstorming outputs

### Solution Capability Matrix
- Maps required capabilities against solution options
- Shows which option addresses which need
- Supports options appraisal and recommendation

## Business Case Components

A complete business case includes:

1. **Problem/Opportunity** — Situation statement, relevant data, affected stakeholders
2. **Analysis of the Situation** — Goals, objectives, root causes, needed vs. existing capabilities, gaps
3. **Recommendation** — Feasibility results, constraints/assumptions/risks, rank-ordered alternatives, cost-benefit analysis, implementation approach with milestones
4. **Evaluation** — Plan for measuring benefits realisation with metrics

### Situation Statement Format
"Problem/opportunity of **[A]** has the effect of **[B]** with the impact of **[C]**."

### Three Levels of Scope Documents

| Document Type | Project Size | When to Use |
|---|---|---|
| **Business Case Document** | Large-scale, enterprise-level impact | Full business requirements, scope, context, CBA, risks |
| **Vision and Scope Document** | Medium/small-scale | Features per release + high-level integration context |
| **Statement of Work (SOW)** | Enhancements, <1 month | One-page business need explanation |

## Current State / Future State Assessment Framework

From the PMI Guide:

1. **Assess Current State** — Using business architecture techniques, capability analysis, process flows, SWOT
2. **Determine Future State** — Using benchmarking, feature models, gap analysis, Kano analysis, purpose alignment
3. **Determine Viable Options** — Using cost-benefit analysis, weighted ranking, real options, valuation techniques
4. **Provide Recommendation** — Ranked options with feasibility assessment

**Proposal language:** "Our approach follows a structured current-state/future-state methodology: Phase 1 maps the as-is environment, Phase 2 defines the to-be vision benchmarked against regional best practice, and Phase 3 develops costed options with a recommended way forward."
