# SaaS Demo Script Template

Discovery-tuned demo script for SaaS implementation and SaaS product-development proposals. Used inside the proposal's Methodology, POC scope, and oral presentation prep — and as a delivery artefact during the engagement (demo design as a paid deliverable).

## Principles

- The demo is not a feature tour; it is the proof block of the diagnosed pain.
- Open with the pain, not the product. Within sixty seconds the buyer should hear their own pain restated.
- Demo only the capabilities tied to the diagnosed pain and decision criteria — every extra feature is a procurement risk.
- Every demo step ends with a "Does this match what you see in your environment?" check — converting demo into ongoing discovery.
- The demo closes with the next step explicitly stated and agreed.

## Demo Script Structure

### 1. Frame (3 minutes)

- Restate the diagnosed pain chain: surface, operational, financial, strategic.
- Name the Critical Event that makes the demo timely.
- Confirm the role of each person in the room and what each needs to see.
- State the demo agenda and the time-box.

### 2. Outcome Preview (2 minutes)

- Show one screen or one number that represents the after-state.
- This is the "wow moment" that earns attention for the next 25 minutes.
- Connect it explicitly to the buyer's metric and baseline.

### 3. Walk-Through, Pain by Pain (3–5 segments of 4–5 minutes each)

For each diagnosed pain, run one segment with this micro-structure:

| Step | What | Owner |
|---|---|---|
| Pain restatement | "You told us X causes Y, which costs Z." | Solution lead |
| Capability shown | The specific feature, flow, or output that addresses X. | Solution lead |
| Real data | Show with realistic data, not lorem ipsum. | Solution lead |
| Match check | "Does this match what you see in your environment?" | Solution lead |
| Capture | Note the response and any new pain it reveals. | Account director |

### 4. Integration Reality (3 minutes)

- Show how the SaaS connects to the existing systems the buyer named in discovery.
- Acknowledge the integration items not yet built — this is integrity, not weakness.
- Reference the integration discovery workstream that will be in Phase 3 of the MAP.

### 5. Trust and Compliance Snapshot (3 minutes)

- Tenant isolation model (silo / pool / bridge) with a single diagram, not a deck.
- Data residency by region.
- Security certifications, sub-processors, BCDR posture.
- Audit and observability posture.

### 6. Commercial Snapshot (2 minutes)

- High-level pricing model and tier ladder.
- Implementation fee logic (driver-based, not lump sum).
- Payment milestones tied to value events, not calendar dates.

### 7. Mutual Action Plan Preview (3 minutes)

- Walk through the Phase 1 and Phase 2 MAP rows.
- Name the next decision and its date.
- Capture any objections raised and convert each into a written follow-up commitment.

### 8. Close (2 minutes)

- "Based on what you saw, what is the most important thing to clarify before our next meeting?"
- "Who else needs to see this demo before the next decision?"
- Confirm the next meeting date and the decision it will support.

## Demo Roles

- **Solution lead** — drives the screens, restates pain, runs the match-check loop.
- **Account director** — captures responses, manages time, runs the close.
- **Domain SME** — present only when the diagnosis requires it; usually silent until invited.
- **Customer success lead** — present from the second demo onwards; this person becomes the post-sale relationship.

## Demo Environment Discipline

- A dedicated demo tenant per opportunity, populated with realistic data for that buyer's vertical and scale.
- No browser tabs, notifications, or unrelated bookmarks visible.
- Pre-flight check 30 minutes before: data refresh, screen resolution, audio, secondary monitor, backup recording.
- A backup recorded demo available for replay if connectivity fails.

## Anti-Patterns

- Feature tour with no pain restatement — the buyer disengages within 5 minutes.
- Demoing for everyone in the room rather than the dominant decision criterion.
- Apologising for what the product does not yet do — instead, name it as a roadmap item and tie to the MAP.
- Closing without naming the next decision and its date.
- Using lorem ipsum, "Acme Bank", or other placeholder data when real-vertical data is available.

## Post-Demo Follow-Up

Within 24 hours, send:

- A one-page summary of the diagnosed pain confirmed in the demo.
- A list of follow-up commitments captured during the demo.
- An updated MAP showing the next decision.
- Any new objections converted into a written objection map (do not address objections in long emails — surface them so they can be discussed properly).
