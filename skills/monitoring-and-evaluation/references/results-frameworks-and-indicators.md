# Results Frameworks, Indicators, and Theories of Change

Reference material for the monitoring-and-evaluation skill. Provides detailed guidance on designing results frameworks, selecting and defining indicators, setting baselines and targets, and constructing theories of change. These are the building blocks of any M&E plan.

Sources: UNDP Handbook on Planning, Monitoring and Evaluating for Development Results (2009), ILO Basic Principles of Monitoring and Evaluation, IFRC Handbook for Monitoring and Evaluation (2002), Seasons (2020), Uzbekistan M&E Guide (2025).

---

## 1. The Results Chain — From Inputs to Impact

The results chain is the backbone of results-based management. It maps the causal pathway from resources invested to long-term change achieved.

```
Inputs → Activities → Outputs → Outcomes → Impact
```

| Level | Definition | Accountability | Example |
|---|---|---|---|
| **Inputs** | Financial, human, and material resources invested | Within project control | Budget, consultants, equipment |
| **Activities** | Actions performed to transform inputs into outputs | Within project control | Training workshops, system development, consultations |
| **Outputs** | Tangible goods produced or services delivered | Within project control | Training manuals produced, system deployed, staff trained |
| **Outcomes** | Changes in behaviour, capacity, or systems resulting from use of outputs | Within project influence | Staff using new system correctly, improved service delivery times |
| **Impact** | Long-term, sustainable changes in conditions | Beyond project control | Reduced poverty, improved health outcomes, increased revenue |

**The critical distinction (UNDP):** Monitoring tracks inputs, activities, and outputs (what we control). Evaluation assesses outcomes and impact (what we influence but do not control). Proposals that conflate outputs with outcomes — "we will train 200 staff" as an outcome — signal weak M&E understanding.

## 2. Theory of Change — Beyond the Linear Chain

### The Problem Tree Method (UNDP)

Before constructing a theory of change, conduct a problem analysis using the problem tree:

1. **Trunk** — write the central problem on the trunk
2. **Roots** — identify root causes below (ask "why?" repeatedly — each cause becomes a branch of roots)
3. **Branches** — identify effects above (ask "so what?" — each effect becomes a branch)
4. **Validation** — verify with stakeholders that the cause-effect relationships are accurate

The problem tree then converts directly into a theory of change by inverting: each root cause becomes an area of intervention, each effect becomes an expected outcome.

### Theory of Change Components

| Component | Content | Quality Test |
|---|---|---|
| **Problem statement** | The specific problem the intervention addresses | Is it evidence-based? Can it be verified? |
| **Context analysis** | The environment in which the change will occur | Does it identify enablers and barriers? |
| **Causal pathways** | How activities lead to outputs, outputs to outcomes, outcomes to impact | Is each link explicitly stated? |
| **Assumptions** | Conditions that must hold for each causal link to work | Are they testable? Are they realistic? |
| **Risks** | Factors that could disrupt the causal pathway | Are they mapped to specific links? |
| **Stakeholders** | Who drives, enables, or blocks each stage of the pathway | Are roles and incentives identified? |

### Theory of Change vs Logframe

| Feature | Theory of Change | Logframe |
|---|---|---|
| **Format** | Narrative + diagram | Matrix / table |
| **Focus** | Explaining *why* the intervention will work | Specifying *what* will be measured |
| **Assumptions** | Central — each causal link has explicit assumptions | Present but often treated as formality |
| **Flexibility** | Adaptive — updated as understanding deepens | Relatively fixed once agreed |
| **Use** | Strategic thinking, programme design, evaluation | Monitoring, reporting, accountability |
| **Best for** | Complex programmes with multiple causal pathways | Donor reporting and compliance |

**Proposal application:** Always present both. The theory of change shows your analytical depth; the logframe shows your measurement discipline. A proposal with only a logframe looks mechanical; a proposal with only a theory of change looks unmeasurable.

## 3. Indicator Design — The CREAM Criteria

Beyond SMART, use the CREAM criteria (UNDP) for indicator quality:

- **C — Clear**: unambiguous definition; anyone reading it interprets it the same way
- **R — Relevant**: directly measures progress towards the intended result
- **E — Economic**: data can be collected at reasonable cost and effort
- **A — Adequate**: provides sufficient information to assess performance
- **M — Monitorable**: data can be collected repeatedly over time using the same method

### Indicator Reference Sheet

For each indicator in the results framework, complete an indicator reference sheet:

| Field | Content |
|---|---|
| **Indicator name** | Clear, specific title |
| **Definition** | Precise description of what is measured and how |
| **Unit of measurement** | Number, percentage, ratio, index, yes/no |
| **Disaggregation** | By sex, age, geography, disability, or other relevant category |
| **Baseline value** | Current value before the intervention (with date) |
| **Target value** | Expected value at project end (with date) |
| **Interim targets** | Milestones (quarterly, annual) to track trajectory |
| **Data source** | Where the data comes from (survey, admin records, reports) |
| **Collection method** | How data is gathered (interview, extraction, observation) |
| **Frequency** | How often data is collected (monthly, quarterly, annually) |
| **Responsible** | Who collects, who analyses, who reports |
| **Limitations** | Known data quality issues or measurement challenges |

**Proposal application:** Include 2–3 sample indicator reference sheets as annexes to demonstrate measurement rigour. Most competitors provide only the logframe matrix; including reference sheets signals professional M&E capability.

## 4. Indicator Types — Process, Outcome, and Progression

### Process (Implementation) Indicators (ILO)

Track whether the programme is being implemented as planned:

| Indicator | Formula | Purpose |
|---|---|---|
| **Composition of participants** | Number of entrants by category / Total entrants × 100 | Verify targeting criteria compliance |
| **Coverage rate** | Number of programme participants / Total target population × 100 | Measure reach of the programme |
| **Implementation rate** | Number of actions implemented / Number of actions planned × 100 | Track pace against plan |
| **Cost per participant** | Total programme cost / Number of participants | Monitor cost efficiency |
| **Drop-out rate** | Number who left before minimum period / Number of entrants × 100 | Signal engagement problems |

### Outcome Indicators

Track whether the programme is achieving its intended results:

| Indicator | Formula | Purpose |
|---|---|---|
| **Gross achievement rate** | Number achieving target outcome / Total participants × 100 | Primary effectiveness measure |
| **Cost per outcome** | Total cost / Number achieving outcome | Cost-effectiveness |
| **Sustainability rate** | Number sustaining outcome at follow-up / Number achieving outcome × 100 | Durability of results |

### Progression Indicators (ILO)

Measure incremental change in the status of beneficiaries who are far from the target state — particularly useful for development programmes:

- Track movement between stages (e.g., from inactive to job-seeking, from job-seeking to training, from training to employment)
- Useful when the final outcome (e.g., employment) may take longer than the programme period
- Allows credit for intermediate progress, not just final achievement

**Proposal application:** Include progression indicators when proposing for programmes targeting marginalised populations. They show evaluators that you understand change is incremental and that you can demonstrate value even when full outcomes take time.

## 5. Baselines, Targets, and Disaggregation

### Setting Baselines (ILO + UNDP)

A baseline is the value of an indicator before the intervention begins. Without a baseline, you cannot measure change.

**Four-step process:**
1. **Identify the indicator** and its precise definition
2. **Determine the data source** — existing secondary data or new primary data collection
3. **Collect the baseline data** — ideally during the inception phase
4. **Document and validate** — record the value, date, source, and methodology

**When baselines are unknown at proposal stage:** State "Baseline to be established during inception phase" rather than guessing. But propose the methodology for baseline data collection — this shows evaluators you have a plan.

### Setting Targets (ILO)

A target is the expected value of an indicator at a specific future date. Four approaches:

1. **Historical trend analysis** — project forward from existing data trends, plus the value the programme adds
2. **Research-based** — use evidence from similar programmes to set realistic expectations
3. **Benchmarking** — compare against high-performing programmes or international standards
4. **Stakeholder negotiation** — agree targets with clients and donors based on what is realistic given resources and context

**Common mistakes:**
- Setting targets without baselines (no basis for measurement)
- Setting targets that are too ambitious (programme cannot achieve them) or too conservative (programme adds no value)
- Using only output targets ("train 200 staff") without outcome targets ("80% of trained staff demonstrate proficiency")

### Disaggregation (ILO + UNDP)

All indicators should be disaggregated by relevant categories to reveal who is benefiting and who is being left behind:

| Category | Why | Minimum Disaggregation |
|---|---|---|
| **Sex** | Gender equality is a cross-cutting requirement for all donors | Male, female, other/prefer not to say |
| **Age** | Youth, working-age, and elderly populations have different needs | 5-year age bands or programme-relevant categories |
| **Geography** | Urban/rural disparities are significant in East Africa | District, sub-county, or urban/rural |
| **Disability** | Disability inclusion is increasingly required (UNCRPD) | Persons with/without disability; Washington Group questions for detail |
| **Education** | Determines capacity and programme design | No formal, primary, secondary, tertiary |
| **Income/wealth** | Poverty targeting verification | Wealth quintiles or income bands |

**Proposal application:** In every logframe and indicator table, include a "Disaggregation" column. This is now a mandatory scoring criterion in World Bank, UNDP, and most bilateral donor evaluations. Omitting disaggregation signals weak GESI awareness.

## 6. Results and Resources Framework (UNDG)

The UNDG Results and Resources Framework (RRF) is the standard tool for UN joint programming. It links results to the resources needed to achieve them:

| Column | Content |
|---|---|
| **National priority** | The government strategy or plan the outcome contributes to |
| **UNDAF outcome** | The collective UN outcome statement |
| **Indicators** | SMART indicators with baselines and targets |
| **Means of verification** | Data sources for each indicator |
| **Risks and assumptions** | What could go wrong and what must hold true |
| **Implementing partners** | Who is responsible for delivery |
| **Indicative resources** | Budget allocated to this outcome |

**Proposal application:** When bidding for UN system work, align your results framework to the RRF format. Show how your assignment contributes to UNDAF outcomes and national priorities. This signals familiarity with UN programming architecture.

## 7. Performance Management Cycle (ILO)

Results-based management is not a one-time exercise — it is a continuous cycle of seven steps:

1. **Formulate objectives** — identify results in clear, measurable terms
2. **Identify indicators** — specify what will be measured for each objective
3. **Set targets** — define expected performance levels with deadlines
4. **Monitor results** — collect data regularly against the indicator framework
5. **Review and report results** — compare actual vs planned; explain variances
6. **Integrate evaluations** — conduct periodic evaluations to answer questions monitoring cannot
7. **Use performance information** — feed findings into decision-making, learning, and accountability

**The "so what?" test (UNDP):** At every stage, ask "So what difference does our intervention make?" If the M&E system cannot answer this question, it is measuring the wrong things.

## 8. Common Indicator Frameworks by Sector

### Governance and Institutional Reform
- Number of policies/regulations adopted
- Percentage of institutions meeting performance standards
- Citizen satisfaction with public services (survey-based)
- Transparency index or access-to-information compliance rate

### ICT and Digital Transformation
- System uptime and availability
- Number of active users / registered users (adoption rate)
- Transaction processing time (before vs after)
- Data accuracy rate
- Cost per transaction

### Capacity Building
- Number of staff trained (output)
- Proficiency assessment pass rate (outcome)
- Percentage applying new skills at 3 months post-training (outcome)
- Performance improvement in trained areas (impact)

### Health Systems
- Service utilisation rates (outpatient visits, immunisation coverage)
- Stock-out rates for essential medicines
- Health worker density per population
- Patient satisfaction scores

**Proposal application:** Select indicators appropriate to the sector and assignment. Sector-specific indicators demonstrate domain expertise. Generic indicators ("number of workshops conducted") suggest the firm does not understand the sector.

## 9. Indicator Quality Assurance Checklist

Before finalising any indicator in a proposal, verify:

- [ ] Is the indicator clearly defined? Would two different people measure it the same way?
- [ ] Is the baseline stated or is a methodology for establishing it proposed?
- [ ] Is the target realistic given the resources and timeline?
- [ ] Is the data source identified and accessible?
- [ ] Is the collection method described and feasible?
- [ ] Is the frequency of collection appropriate (not too frequent, not too infrequent)?
- [ ] Is a responsible person or team identified?
- [ ] Is disaggregation specified (at minimum by sex and geography)?
- [ ] Does the indicator measure change, not just activity?
- [ ] Can the indicator be influenced by the project (attribution or contribution)?
